# Main Finding

The game has strong underlying systems, but the player-facing loop needs to be connected more clearly.

Priority should be:

```text
Opening / mission selection
→ encounter selection
→ real battle
→ victory rewards
→ progression update
→ summon / unlock
→ return to kingdom
```

The goal is to make the first 5 minutes feel complete.

---

# Highest Priority Fixes

## 1. Wire Encounter Cards Into Real Battles

Current issue:

```text
The shift / encounter screen appears to show encounters, but tapping encounter cards may not reliably start a real battle.
```

Goal:

```text
When the player taps an encounter card, it should start the correct battle in battle.tsx using the selected enemy/encounter data.
```

Expected behavior:

```text
1. Player sees available encounters.
2. Player taps Air Sprite or another encounter.
3. App opens the battle screen.
4. Battle screen loads the correct enemy.
5. Enemy clues, HP/corruption, system type, and rewards match the selected encounter.
```

Do not hardcode only one enemy if the existing structure supports multiple enemies.

If necessary, pass encounter ID or enemy ID through route params or existing state.

---

## 2. Connect Boss Battle Button to a Real Boss Fight

Current issue:

```text
The boss screen appears to have lore and an Enter button, but it may not connect to an actual boss battle.
```

Goal:

```text
When the player taps Enter on the boss screen, the app should launch a real boss battle using existing battle.tsx logic.
```

Expected behavior:

```text
1. Player opens boss screen.
2. Player sees boss briefing.
3. Player taps Enter.
4. Battle screen opens with boss enemy data.
5. Boss has appropriate corruption/stability pressure, clues, rewards, and victory result.
```

Use existing battle systems rather than creating a separate boss battle engine.

---

## 3. Strengthen Victory Rewards and Progression Loop

Current issue:

```text
Battle rewards and progression may not feel complete or visible enough.
```

Goal:

```text
After battle victory, show a clear reward/progression screen.
```

The victory screen should show:

```text
Victory title
Enemy defeated
Region restored progress
Player EXP
Hero EXP
Codex Shards
Items or materials if applicable
Stars earned
Next suggested action
```

Example:

```text
Victory: Air Temple Restored

You recognized the breathing clues, stabilized the patient, used an Air intervention, and reassessed.

Rewards:
Player EXP +30
Novice Guardian EXP +25
Codex Shards +45
Air Region Progress +1

Air Temple: 1/5 Restored
```

Buttons:

```text
Continue
View Codex
Return to Kingdom
Next Mission
```

Do not make the rewards overly complex yet.

---

## 4. Make Region Progress Visible

Goal:

```text
Make the player clearly see that winning battles restores body-system regions.
```

Example:

```text
Air Temple
Respiratory System
1/5 Restored
Next: Wheezing Woods
```

If region progress already exists in player state, use it.

If not, prepare a simple region progress structure:

```js
player.regionProgress = player.regionProgress || {
  Air: 0,
  River: 0,
  Fire: 0,
  Energy: 0,
  Protection: 0,
  Mind: 0,
  Storm: 0,
  Forge: 0,
  Filter: 0,
  Growth: 0
};
```

Do not overbuild the kingdom map yet. A simple progress display is enough.

For each condition, show:

```text

Real medical concept
System region
Common clues
What the player should recognize
Why certain actions worked
```

Example:

```text
Air Sprite Corruption

Real Concept:
Asthma Exacerbation / Bronchospasm

System:

Clues:
Breathing Fast
O2 Glow Dim
```

# MongoDB / Backend Guidance

MongoDB is not the top priority right now.

Current understanding:

```text
AsyncStorage allows local/offline play.
MongoDB is mainly for cloud save/cross-device sync.
MONGO_URL may currently be missing.
```

For now:

```text
Do not block gameplay because MongoDB is missing.
Keep local save working through AsyncStorage.
If backend sync fails, fail gracefully without interrupting gameplay.
```

Later:

```text
Add MONGO_URL as a Replit Secret.
Improve sync conflict handling.
Move reward validation server-side if monetization becomes important.
```

Do not prioritize MongoDB over completing the first playable loop.
